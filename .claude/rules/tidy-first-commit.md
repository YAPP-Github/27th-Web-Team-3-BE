# Tidy First 커밋 전략

구조적 변경과 행동적 변경을 분리하여 커밋합니다.

## 커밋 분류

### 구조적 변경 (Structure)

행동 변경 없이 코드 구조만 변경합니다:

- 파일/디렉토리 생성, 이동, 삭제
- 함수/타입 이름 변경
- 모듈 분할/병합
- import 경로 변경
- pub → pub(crate) 변경

**커밋 메시지 형식**:
```
🏗️ structure: [변경 내용]

- 세부 변경 1
- 세부 변경 2
```

**예시**:
```
🏗️ structure: prompt.rs를 prompt/ 디렉토리로 분할

- prompt/mod.rs 생성
- prompt/guide.rs 분리
- prompt/refine.rs 분리
- prompt/examples.rs 분리
```

### 행동적 변경 (Behavior)

기능이나 로직을 변경합니다:

- 새 기능 추가
- 버그 수정
- 검증 규칙 변경
- API 응답 형식 변경

**커밋 메시지 형식**:
```
✨ feat: [기능 설명]
🐛 fix: [버그 설명]
♻️ refactor: [리팩토링 내용]

- 세부 변경 1
- 세부 변경 2
```

## 커밋 순서 원칙

1. **먼저 구조적 변경** → 테스트 통과 확인
2. **그 다음 행동적 변경** → 테스트 통과 확인
3. **절대 같은 커밋에 혼합하지 않음**

## 예시 워크플로우

### 좋은 예

```bash
# 커밋 1: 구조 변경
git commit -m "🏗️ structure: client.rs 분리"

# 커밋 2: 행동 변경
git commit -m "✨ feat: 타임아웃 설정 추가"
```

### 나쁜 예

```bash
# 한 커밋에 구조+행동 혼합 (금지)
git commit -m "client.rs 분리 및 타임아웃 추가"
```

## 장점

- 리뷰어가 변경 의도를 명확히 파악
- 구조 변경은 자동 병합 가능
- 문제 발생 시 롤백 범위 최소화
- 코드 히스토리 추적 용이
